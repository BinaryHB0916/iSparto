Wave 4 has a single primary objective and three supporting sub-objectives:

1. **Primary — make the guardian enforceable.** Wires `scripts/language-check.sh` into the `/end-working` Doc Engineer audit such that a non-zero exit code from the guardian blocks Lead from proceeding to `gh pr create`. 2. **Sub-objective A — role definition sync.** `docs/roles.md` > "Doc Engineer" audit checklist needs a new item 9 that describes exactly how to invoke the guardian and how to interpret its three exit codes (0/1/2), and the output template needs a slot for the violation report on FAIL. 4. **Sub-objective C — bounded recovery.** When the loop bound is hit, there must be a deterministic, non-improvised recovery path — not "Lead figures it out". The Wave 3 IR left a forward note about this. The path must preserve WIP (push the branch), document the blocked state (plan.md entry), inform the user, and exit cleanly without creating or merging a PR.

One important non-objective for Wave 4: it is NOT expected to translate any new files, introduce any new detector rules, or modify the guardian script itself. The script is already at 0/0/0 against the post-Wave-3 tree. Wave 4 is pure plumbing — command template + role definition + plan.md housekeeping.
