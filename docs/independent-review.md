### My understanding of the Wave intent (from plan.md + CLAUDE.md "Documentation Language Convention")

The Wave 2 product intent has two layers:

1. **Mechanical layer (CJK removal):** After Wave 2, every Tier 1 file must be free of CJK characters so the guardian, when promoted to a blocking gate in Wave 4, does not block any existing Tier 1 file.
2. **Principle layer (no hard-coded user-facing strings):** CLAUDE.md's Documentation Language Convention states that Tier 1 files "must not embed literal user-facing strings in any specific language. Describe the intent in English and let the Lead generate the actual string in the user's language at runtime." The Wave 2 translation must therefore not simply swap Chinese literals for English literals; it must convert direct output strings into "Inform the user (in user's language) that …" intent-descriptions (the `illustrative-example rule` immediately after). Any English literal that survives as a user-facing output string in a Tier 1 file is still a violation of this rule — the rule is language-agnostic.

A future wave (Wave 4 or earlier) should extend `language-check.sh` with a heuristic for literal English quoted strings inside instruction files, or at minimum add a human audit item in the Doc Engineer checklist for "Tier 1 literal user-facing strings".
