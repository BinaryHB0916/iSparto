# Core Concepts

> If you're familiar with software team collaboration, the following concepts can be quickly understood by analogy.

## Wave Parallelism

**The entire premise of Wave parallel execution is that tasks are fully decoupled. This is the framework's most critical concept.** This applies to both write tasks (coding, documentation) and read tasks (code review, doc audit, research/debug). If two tasks touch the same file, or if A's output is B's input, they cannot be placed in the same Wave for parallel execution.

The Team Lead's core job when breaking down tasks is not "distributing work," but **determining which tasks can run simultaneously and which must run sequentially**. The criteria are:

- **File level**: Do the two tasks modify overlapping files? If so, they cannot run in parallel.
- **Data level**: Does A produce a data structure that B consumes? Then A must finish before B can start.
- **Logic level**: Do the two features have runtime dependencies? For example, login must be completed before payment can use the user session.
- **Context capacity**: Can a single Developer's full input (task description + code files in scope + relevant spec sections + interface contracts) fit in one context window? If not, split the task into smaller pieces or move parts to a later Wave. This is a practical ceiling — even perfectly decoupled tasks fail if the Developer cannot hold all necessary context at once.

**Tasks that can be decoupled go into the same Wave for parallel acceleration; tasks that cannot be decoupled are split into different Waves for sequential execution.** This is the underlying logic of "parallel within a Wave for speed, sequential across Waves for quality."

Two tools are used to achieve decoupling:
- **File ownership** — Each Developer can only modify files within their assigned scope, physically preventing conflicts
- **Interface contract** — When multiple Developers' code needs to integrate, interfaces are defined upfront, each develops against the contract, and integration happens at the end

If the Team Lead finds that tasks within a Wave cannot be fully decoupled, the correct approach is to **split them into smaller Waves**, rather than forcing parallelism at the risk of conflicts.

## Runtime Output Layering (A/B/C)

**iSparto treats the user as a decision-maker, not an observer.** Every user-facing output the Lead emits at runtime must be classified into one of three layers *before* emission, and the classification is mechanical (5 enumerated trigger types), not stylistic.

- **A-layer — decision interruption.** Blocks the user and demands a response. Restricted to 5 mechanically-identifiable trigger types: (a) Lead proposes a new plan, (b) Codex surfaces P0/P1 findings, (c) irreversible operation imminent, (d) IR requests deep-IR script authorization, (e) Process Observer critical intercept. Every A-layer output uses a fixed wording template: "I plan to X, because Y. If you disagree, I can switch to Z. Continue?" — never a multiple-choice menu.
- **B-layer — decision preparation at natural pause points.** Emitted only at three fixed points: `/start-working` opening briefing, `/end-working` closing briefing, `/plan` proposal step. Does not block. Contains the cross-session recovery surface (current Wave status, remaining issues, next task) as protected content — these are state variables the user needs to resume work, not implementation noise. Command templates pin the B-layer briefing shape to a fixed structure so Lead's runtime judgment narrows to word choice.
- **C-layer — silent archive.** Never emitted to the user; logged to `docs/session-log.md`, commit history, or hook logs where the user can grep when they care. C-layer is where the framework's internal housekeeping lives: branch auto-create, hook verification green, `gh auth switch`, health check green, Process Observer armed, Doc Engineer idle, auto-repair migrations. The framework does "everything is green" as silence, not as a status message.

Why layering matters: before v0.7.4, Lead's facts-dumping pattern ("Branch X. Hooks Y. gh Z. Plan W. Memory V. Ready to proceed.") forced the user to re-do the Lead's job — sort signal from noise, figure out which item needed a response, and which item was just the machine running correctly. Layering moves that work back to where it belongs: the Lead classifies, the template enforces, and the user sees only decisions and decision-preparation. See `docs/design-principles/information-layering-policy.md` for the full Policy (7 principles, enforcement, conflict resolution, revision rule) and `docs/design-principles/conversation-style.md` for the wording rules and before-after samples.

## Concept Quick Reference

| Concept | Explanation | Analogy |
|---------|-------------|---------|
| **Wave** | A batch of decoupled tasks. Tasks within a Wave execute in parallel, Waves execute sequentially, and users validate at Wave boundaries. | Similar to a Sprint, but lighter and more granular |
| **File ownership** | When the Team Lead breaks down tasks, they assign each Developer a specific set of files they can modify, physically isolating parallel tasks. | Similar to Git CODEOWNERS, but dynamically assigned per task |
| **Interface contract** | When multiple Developers work in parallel, the Team Lead predefines function signatures, parameter types, and return values between modules, ensuring independently developed code can integrate. | Similar to API documentation, but defined before development |
| **MCP (Model Context Protocol)** | A protocol defined by Anthropic that allows Claude Code to call external tools. iSparto uses MCP to invoke Codex for code review and QA. | Similar to a plugin system |
| **tmux teammate mode** | The built-in Agent Team execution mode in Claude Code. Each Teammate runs in its own tmux Claude Code session, carrying the full context of one branch (scoped codebase reading, Developer prompt assembly, Developer output review). Lead's main session stays pure governance and holds only the decomposition result, Teammate progress, and interface contracts needed for final merge. Teammates exist for **context isolation**, not for concurrency workarounds — Codex MCP natively supports concurrent threadId-scoped sessions. | Similar to multiple terminal windows working simultaneously |
| **Agent Team** | An experimental feature of Claude Code that allows a primary session (Team Lead) to launch multiple sub-sessions (Developers) for parallel development. | Similar to a project manager leading multiple programmers |
| **Independent Reviewer** | Product-technical alignment verifier spawned via Codex CLI in a tmux pane (GPT-5.4) with zero inherited context — cross-provider blind review. Triggers at Phase 0, Wave Boundary, and A-layer interrupts. Most token-intensive role per invocation — a deliberate tradeoff for genuine independence. | Blind peer reviewer in academic publishing — reads the paper fresh, without the authors' cover letter |
| **Snapshot/Restore** | Automatic pre-change backup. Created before `/init-project`, `/migrate`, and `install.sh`. `/restore` rolls back to any snapshot. Stored in `~/.isparto/snapshots/`. | Similar to a system restore point |
| **Session Log** | Automatic development metrics collected by `/end-working` and reported by `/start-working`. Stored in `docs/session-log.md`. Tracks: tasks completed, developers spawned, Codex reviews, files changed, key decisions. | Similar to a daily standup summary |
| **Process Observer** | The compliance supervision role in the team. Composed of two parts: real-time interception (PreToolUse hooks block dangerous operations before execution) and post-session audit (sub-agent reviews workflow compliance during /end-working). Does not participate in development decisions — only monitors process compliance. | Similar to a CI/CD policy check + post-sprint retrospective |
| **Compliance Audit** | The post-session review performed by Process Observer during /end-working. Checks 6 categories (branch conventions, code review triggers, Doc Engineer execution, PR workflow, out-of-scope operations + plan.md contract, independent reviewer) across 19 items. Outputs a deviation report with PASS/FAIL/WARNING status. | Similar to a SOX compliance audit, but for development workflow |
| **User Preference Interface** | The boundary between Claude Code auto-memory (user's personal preferences) and CLAUDE.md (agent team's workflow rules), governed by the **territory principle**: memory = user space, CLAUDE.md = agent team space. Memory is read-only input — the team adapts communication style but never overrides workflow rules. Three levels: unconditional (language, style), conditional (pace, autonomy — within workflow bounds), record-only (process changes — workflow wins, user informed). | Similar to company policy vs personal work style — you can wear casual clothes (preference) but you can't skip code review (policy) |
| **Module Memory** | Per-module record of technical decisions and implementation lessons, stored in the "Decisions & Lessons Learned" section of tech-spec.md. Survives across sessions and prevents repeated mistakes within a module. Part of a three-layer memory model: Project memory (CLAUDE.md rules + plan.md), Module memory (tech-spec.md per-module decisions), Personal memory (Claude Code auto-memory for user preferences). | Similar to a team wiki page per component — "things we learned building the auth module" |
| **Behavioral Template** | A Markdown file that functions as a system prompt or structural blueprint for AI agent behavior. `commands/*.md` (slash command definitions) and `templates/*.md` (document generation blueprints) fall into this category. Despite the `.md` extension, changes to these files alter how agents act and go through Developer review (Tier 2b) — they are NOT "pure documentation" under the Tier system. | Similar to Ansible playbooks or Terraform configs — text format but drives automated behavior |
| **Stateless Session** | Each iSparto session starts with zero memory — all state is reconstructed from external artifacts: plan.md (task state), session-log.md (history), git log (code changes), and CLAUDE.md (rules). This is why documentation sync is non-negotiable: docs ARE the agent's memory. If something isn't written down, it doesn't exist for the next session. Inspired by the "Stateless Reducer" principle from 12 Factor Agents. | Similar to a REST API — each request carries all the context it needs; the server retains nothing between requests |
| **Runtime Output Layering (A/B/C)** | Every user-facing output is classified before emission into A-layer (decision interruption, 5 trigger types), B-layer (decision preparation at `/start-working` / `/end-working` / `/plan` pause points with protected cross-session recovery surface), or C-layer (silent archive, logged but never emitted). Enforced structurally via fixed B-layer briefing shapes in `commands/*.md`. See `docs/design-principles/information-layering-policy.md` for the full Policy. | Similar to a pilot pre-flight checklist — the pilot doesn't narrate every switch flip to the passengers, only the decisions that need passenger awareness |
| **A-layer Peer Review** | Before Lead emits any A-layer interrupt (decision-blocking output), the Independent Reviewer independently validates the A-vs-B-vs-C classification — "is this really decision-worthy, or should it be B-layer briefing / C-layer silence?" Prevents Lead from over-interrupting the user with facts that don't need a decision. See [docs/design-principles/a-layer-peer-review.md](design-principles/a-layer-peer-review.md) for the full protocol. | Similar to a second pair of eyes on outbound customer communication — a colleague checks whether the email really needs to interrupt the client or can wait for the weekly update |
| **Guardian Scripts** | Five workflow guardrail/helper scripts invoked at `/start-working` and `/end-working` boundaries: `language-check.sh`, `policy-lint.sh`, and `plan-md-contract-check.sh` block commit-path violations (Tier 1/2 CJK + Principle 1 / session-log ceremony / plan.md-session-log-CHANGELOG separation respectively); `gh-account-guard.sh` guards PR account attribution immediately before `gh pr create`; `session-health.sh` renders the `/start-working` session-opening B-layer health preview. Enumerated in `CLAUDE.md` Module Boundaries. | Similar to CI guardrails scoped to workflow-shape (audit ordering, TODO homing, account drift) rather than code style |
| **Observation-period Tracker** | A 5-row table in `docs/plan.md` (Waves 0–4 post-v0.8.0 upgrade) that records per-Wave DocEng strictness / Lead escalation / Teammate literalization patterns. Release Gate condition 2 requires 3 real Wave Boundary IR passes in this window before `scripts/release.sh` is allowed to ship. Retired once the window closes. | Similar to a canary deployment window — watch 5 production uses of a major upgrade before declaring it trusted and rolling forward |
