# Design Decision Log

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Remove Claude Reviewer | Codex as the sole reviewer | Same-source review has limited value; cross-source review covers blind spots more effectively |
| Codex architecture pre-review | Intervene at Phase 0 | Catching architecture issues before coding begins costs far less than rework after development |
| Codex role positioning | Developer (implementation engine) | Implements code per Lead's structured prompts; Lead reviews output. Cross-model review (Opus reviews Codex output) covers each model's blind spots |
| Team Lead handles prompt assembly | Team Lead assembles structured prompts for Developer | The user does not participate in copy-pasting between roles; Team Lead orchestrates the prompt→Developer→review loop |
| Doc Engineer is a sub-agent | Spawned by Team Lead | Requires Team Lead's global context; does not need an independent tmux pane |
| Developer includes unit tests | Written alongside the code | Tests are part of the code, not a separate phase |
| Platform-agnostic design | Template is not tied to any tech stack | Works for iOS / Android / macOS / Windows / Web / cross-platform |
| Project-specific plugins at project level | Not in global settings.json | swift-lsp is only useful for iOS and should not pollute other projects |
| Unified document naming | All use the -spec suffix | product-spec, tech-spec, design-spec — immediately clear |
| Added tech-spec.md | Separate tech spec document | Separates product behavior (product-spec) from technical implementation (tech-spec); template cost is zero — skip it if the project doesn't need it |
| Team Lead authorization & escalation mechanism | Team Lead decides routine matters independently; escalates when uncertain | Reduces user involvement in intermediate coordination while ensuring critical decisions are not bypassed — better to escalate too much than too little |
| Team Lead document change permissions | Team Lead can modify all documents; reports changes in summary afterward | Team Lead + Doc Engineer are document managers; prohibiting document changes creates bottlenecks. User reviews afterward; product decision changes are marked with a warning for special attention |
| Unified Codex configuration | xhigh reasoning (where MCP supports it) | `codex` tool supports reasoningEffort: xhigh; `review` tool uses server defaults (no reasoningEffort parameter). Fast mode is not exposed by the MCP server and cannot be set via MCP calls |
| Codex QA smoke testing | QA step added after code review | Fills the gap of the QA role in human teams; incremental testing strategy (only test changed paths) addresses the slowness of smoke testing |
| Prerequisites for Wave parallelism | Tasks must be fully decoupled | No file overlap + no data dependencies + no runtime dependencies required for parallelism; if decoupling is not possible, defer to the next Wave — do not risk conflicts |
| Hotfix follows the full workflow | No simplified version | The full Agent Team workflow takes minutes; there is no human-team bottleneck of waiting for people. Hotfixes are precisely when second incidents are most likely, so review should not be cut |
| Project name | iSparto | Greek mythology Spartoi (sow dragon teeth, grow an army) + moved i from the end to the front = I = one person. An army of one |
| Inspired by gstack | Only adopted the product review concept | The plan-ceo-review philosophy is good; /browse and /qa are Web-specific and not universal |
| Effort level | max | Max subscription tokens go unused otherwise; pursue the highest reasoning depth |
| Cost | $120/month | Claude Max $100 + ChatGPT $20 — two top-tier models with no additional fees |
| Memory granularity | Milestone level | Space is limited; details are managed by plan.md |
| Document layering | Concise README + detailed docs/ | Keep README under 200 lines so new users can grasp core information in 30 seconds; in-depth content is split by topic into docs/ |
| Architecture diagrams use mermaid | Replaces ASCII art | CJK character widths are imprecise in GitHub code block fonts, making ASCII borders inevitably misaligned; mermaid renders as SVG and has no such issue |
| macOS + iTerm2 only | No cross-platform terminal adaptation | Agent Team tmux mode depends on iTerm2's built-in tmux integration; focus on the core experience at this stage without spreading effort thin |
| templates/ as a separate directory | Templates extracted from README | Avoids duplicating template content in the README (originally 400+ lines); templates as standalone files can be directly referenced by /init-project |
| Snapshot/Restore system | Automatic snapshot before every destructive operation (install, migrate, init-project) | Git doesn't cover uncommitted state or global ~/.claude/ files; automatic snapshots give users a safety net without requiring any manual action |
| Session Log | /end-working auto-generates session metrics to docs/session-log.md; /start-working reads history | Dogfooding showed manual metric collection is unrealistic; auto-collection provides project health visibility across sessions without user effort |
| Version tracking and upgrade | VERSION file + CHANGELOG.md + install.sh --upgrade with "what's new" display | Users who installed via curl need a clear way to know what version they have and what changed; CHANGELOG displayed inline during upgrade reduces friction |
| Thin bootstrap installer | bootstrap.sh (thin entry) + versioned install.sh (from GitHub Releases) + local isparto.sh stub | Old git-clone approach could not self-update from old versions; release-based install enables SHA256 checksum verification, version pinning, and eliminates .git overhead. Uninstall/restore remain fully offline via the local stub |
| User Preference Interface | Territory-based boundary: memory = user space (preferences), CLAUDE.md = agent team space (workflow rules); three-level model (unconditional / conditional / record-only); memory is read-only input | Claude Code auto-memory can generate behavioral rules that overlap with CLAUDE.md. Without explicit boundaries, memory rules may silently contradict workflow rules. Territory principle provides an objective criterion (topic ownership) rather than subjective conflict detection |
| Plan Mode auto-trigger | Lead autonomously enters plan mode for cross-module, core design, user-facing, or hard-to-reverse changes | Same principle as Solo/Agent Team auto-selection — Lead makes criteria-based decisions without waiting for user instruction. Reduces wasted effort from misaligned implementations |
| Role swap: Opus → prompt assembler, Codex → Developer | Codex implements, Opus reviews | Codex produces higher-quality code with structured prompts; Opus's strength is context understanding and review, not bug-free implementation |
| Role-model decoupling | Centralized configuration table in docs/configuration.md | Roles define responsibilities only; model names live in one table. Changing models only requires updating the table + Doc Engineer audit |
| QA is not a separate role | Developer handles QA via different prompt template | Same model, same MCP, same auth — only the prompt differs. Trigger table defines when to invoke QA mode |
| Acceptance script pre-definition | QA tests against plan.md acceptance scripts, not ad-hoc prompts | "Post-hoc test generation" translates implementation into tests without verifying behavior. Pre-defined action/eval steps make QA deterministic and let users validate "what counts as done" before development starts. Inspired by Wang Wei (@onevcat) and his talk "The Way and the Craft of AI Agents" |
| Context capacity as 4th Wave split criterion | Added alongside File / Data / Logic levels | Decoupling ensures tasks don't conflict, but doesn't ensure tasks fit in a single Developer's context window. A perfectly decoupled task can still fail if the Developer must hold more code + specs than the model can process at once — context overflow causes silent quality degradation |
| Module Memory location | Stored in tech-spec.md "Decisions & Lessons Learned" section, not a separate directory | tech-spec.md already organizes content by module; adding a section there keeps module decisions co-located with module specifications. A separate directory would create a parallel structure that diverges from tech-spec and adds maintenance overhead |
| Three-layer memory model | Project (CLAUDE.md + plan.md) / Module (tech-spec.md) / Personal (Claude Code auto-memory) | Each layer has a clear owner and lifecycle: project memory governs team behavior, module memory preserves implementation lessons, personal memory stores user preferences. Maps to the existing territory principle and adds the missing middle layer for module-scoped knowledge |
| Process Observer layering | Hooks = core layer (hard interception), sub-agent = advisory layer (best practices) | The sub-agent depends on the Lead actively spawning it, so structurally there is no execution guarantee. Critical compliance checks are pushed forward into the un-bypassable Hook layer, and the sub-agent is downgraded to providing improvement suggestions |
| Hook registration layering | User-level = Bash safety rules, project-level = Edit/Write/Codex workflow rules | Edit/Write interception is an iSparto-specific rule, so placing it at user level would affect non-iSparto projects. Bash safety rules are universal, so placing them at user level lets a single upgrade take effect globally. The `/start-working` verification step solves the project-level propagation problem |
| Process Observer Audit uses Sonnet | Sub-agent downgraded to Sonnet 4.6 | The Hooks core layer already covers every critical compliance check, and the sub-agent is the advisory layer. The downgrade does not introduce a human-fallback dependency, because any quality degradation is structurally backstopped by Hooks rather than by a person |
| MCP server rename: codex-reviewer → codex-dev | Resolved | The original name came from the early stage when Codex acted as the reviewer; once the role flipped to Developer, the name no longer matched reality. The fix was a rename plus an install.sh upgrade that auto-migrates legacy MCP registrations for existing users plus /start-working auto-migrating the project-level hook matcher. Decoupling the role from the MCP server name is still limited by the Claude Code hook mechanism (`mcp__<server>__<tool>` is generated automatically from the registration name and cannot be abstracted into a variable), but at least the name is no longer misleading |
| Rejected-approach tracking | A "Rejected Approaches" table in plan.md | Prevents the AI from repeatedly trying paths that have already been disproven. Inspiration: Pretext (chenglou/pretext) explicitly records "tried and rejected" approaches in AGENTS.md. Stored inside plan.md rather than a standalone file, because /start-working always reads plan.md — a standalone file would risk being ignored |
| Behavioral template classification | commands/*.md and templates/*.md are handled as Tier 2b (Developer review, no QA triggered) and are not classified as Tier 3 | These files are system prompts that drive Agent behavior; the .md extension does not mean they are "pure documentation". Tier 1 would be too heavy (prompt iteration is frequent), and Tier 2b strikes a balance between correctness review and iteration speed. The Hooks layer correctly lets .md through (an extension-based check cannot distinguish semantics); semantic classification is enforced at the Tier layer and the Process Observer B1 audit layer |
| Centralize the "no direct code writing" rationale | CLAUDE.md Collaboration Mode defines the single motivation statement; the other 6 sites only reference it (CLAUDE-TEMPLATE.md is the exception — it copies the motivation in full, because it generates a standalone project-level CLAUDE.md and must not retain references that point back inside the framework) | An audit found the rule scattered across 7+ sites while the motivation lived in design-decisions.md alone. The initial fix proposed duplicating the motivation 7 times — rejected, because that is the same anti-pattern as scattering model names across multiple locations. A single definition plus references avoids having to edit N places when the model or the quality reasoning evolves. Note: when the wording of the motivation changes, both CLAUDE.md and CLAUDE-TEMPLATE.md must be updated in sync |
| Self-verifying startup | /start-working adds a build/test run verification after the documentation checks (non-blocking, notification-only rather than a gate) | Inspiration: the Anthropic long-running agent harness runs a smoke test at the start of every session to confirm environment health. Previously iSparto only performed documentation-layer drift checks and never runtime verification, so issues such as dependency updates, build-environment changes, or unsynced changes left over from the previous session went undetected. The check is optional (it depends on whether build/test commands are defined in CLAUDE.md). It is intentionally non-blocking (failures are reported but not intercepted), because some projects have long builds (an iOS build can take 5 minutes) and blocking would slow startup down; this also stays consistent with the overall briefing model of /start-working — notify the user and let the user decide |
| Make the stateless-session principle explicit | Declare the session's stateless design in concepts.md | iSparto had already implemented stateless sessions via plan.md + session-log.md + git, but had never explicitly stated the principle. Declaring it helps users understand why documentation sync cannot be skipped — the documentation is the agent's only memory. Source: 12 Factor Agents (HumanLayer), Factor 12: Stateless Reducer |
| Security-audit three-layer defense | L1 real-time Write/Edit interception + L2 pre-commit scan + L3 /security-audit full audit | Pre-commit alone is not enough, because between the moment a secret is written to a file and the moment the commit happens, other operations can already leak it. L1 intercepts 5 critical patterns in the PreToolUse hook (a hot-path subset only), L2 runs a full scan before commit, and L3 covers history and dependencies. Each layer makes its own performance/coverage trade-off and acts as a backstop for the others |
| security-patterns.json as single source of truth | All security scans share one JSON rule library | Avoids having patterns scattered across scripts and JSON files with each copy maintained separately. Both pre-tool-check.sh and pre-commit-security.sh read from the JSON, so adding or removing a pattern only requires editing one place. The realtime_critical field selects a subset to satisfy the L1 performance constraint |
| Division of labor between security-patterns.json and dangerous-operations.json | dangerous-operations = operational safety (command interception), security-patterns = data safety (content scanning) | The former intercepts dangerous operations (force-push, rm -rf, reset --hard), and the latter scans file contents and staged files. The responsibilities are cleanly separated and do not overlap |
| settings.json rename | settings.json → settings.example.json | A bare settings.json at the repository root is easily mistaken for an active project configuration file; appending the .example suffix makes its role as a reference template unambiguous |
| pre-tool-check.sh JSON parsing limitation (known limitation) | extract_json_field uses awk character-by-character parsing and does not support \uXXXX Unicode escapes or nested JSON objects | Current usage sites (the command, file_path, and prompt fields) are all flat strings with no risk of triggering this limitation. If a future Claude Code tool_input structure change introduces nesting or Unicode escapes, the parser must be switched to Python JSON parsing (install.sh already depends on Python 3) |
| Sensitive-file detection ownership | Removed from dangerous-operations.json and handed over to the security-patterns.json three-layer system | The old rule did substring matching against the command string, so any commit message containing the literal ".env" produced a false positive. The new system scans actual staged files and their content, which is more precise; the separation of responsibilities also avoids duplicate interception and rule conflicts |
| Full-stack coverage of sensitive_files | 7 categories with ~50+ patterns | iSparto is a general-purpose framework and must cover the full set of iOS/Android/macOS/Windows/Web/infrastructure/data-engineering scenarios; the Claude Code source-map incident proved that build artifacts are a real attack surface |
| Block *.map by default | At the sensitive_files level (BLOCK) | A source map contains the full source code (in the sourcesContent field), so committing one is equivalent to leaking the entire codebase; projects that need to keep them go through the .secureignore allowlist |
| *.log not added to sensitive_files | False-positive rate too high | Many projects commit logs to the repository for debugging or CI purposes; the secrets content scan (L2 content scan) already covers any secrets that appear inside logs, so filename-based interception offers little additional benefit |
| Build output directories not added to sensitive_files | Only present in gitignore_baseline | Directories such as dist/ and build/ do not contain secrets themselves; they merely should not be committed to the repository, so .gitignore coverage is sufficient and a hard pre-commit block is not required |
| Do not expand L1 | Keep the 5 critical patterns | Build artifacts are a file-level problem rather than a content-level problem, so L1 Write/Edit interception is not applicable; L2 filename matching is sufficient |
| Add inline source maps to secrets | Caught by the content scan | Some projects do not emit a standalone .map file and instead inline the source map as a data URI inside the bundle; the content scan is required to cover this gap |
| /end-working branch guard | Check the current branch before commit and auto-create a docs/ branch when on main | Step 6 of the /end-working template assumes the commit happens on a feature branch, but once the main work has been merged, committing the session log on main collides with the push-on-main hook. The branch guard resolves this timing-assumption issue |
| Formalize the docs/ and release/ branch prefixes | Added to the CLAUDE.md branch rules and the Process Observer A1/A3 checklist | They were already being used in practice (docs/ for session logs, release/ for releases), but the rules had never been written down, which caused the Process Observer audit to report false positives. The Codex review during the end-working fix flagged this inconsistency |
| Independent Reviewer role | Teammate (tmux) rather than sub-agent | A sub-agent inherits conversation fragments from the Lead (including any biases and incorrect assumptions), whereas a Teammate is a brand-new session with natural context isolation, analogous to blind peer review in academic publishing |
| Independent Reviewer reading order | product-spec → tech-spec (mandatory) | Reading the technical proposal first anchors the reviewer to that proposal; reading the product requirements first is what allows an independent judgment to form |
| Independent Reviewer spawn prompt | A fixed one-liner, with no additional description allowed from the Lead | Any description supplied by the Lead can carry bias; once "only listen to your own voice" was reinterpreted by the Lead as "a binary gate", any description the Lead added would have biased the reviewer in the same direction |
| Independent Reviewer output | Written directly to a file, not filtered through the Lead | The Lead might soften or reinterpret the reviewer's findings; writing directly to a file ensures the user sees the original judgment |
| Developer two-tier model | gpt-5.3-codex (implementation) + gpt-5.4-mini (QA / quick fixes) | Benchmark data: 5.3-codex leads 5.4 on Terminal-Bench at 77.3% versus 75.1% (pure coding specialization). 5.4's advantages (1M context, computer use) are not exercised in the Developer's structured-prompt scenario. The two-tier mapping aligns with the Tier classification, giving a precise quality-versus-speed balance. Cost is unchanged: ChatGPT Plus at $20 includes every model. gpt-5.3-codex-spark validation failed (it is not supported by ChatGPT Plus authentication), so gpt-5.4-mini is used to cover the quick-fix scenario |
| Keep Developer on gpt-5.3-codex rather than upgrading to 5.4 | gpt-5.3-codex remains the Developer default | 5.4 is a general-purpose model that absorbs coding capability, but the iSparto Developer role is defined as "implement code from a structured prompt" — and in pure coding scenarios 5.3-codex is still stronger. Both Reddit community evaluations and Cursor's internal benchmarks confirm that 5.3-codex is stronger inside the rapid iteration-implementation loop. Risk tracking: if 5.3-codex is retired (compare 5.2 Thinking, retired on June 5), a passive upgrade to 5.4 will be required along with verification that prompt templates remain compatible |
| gh account auto-alignment | Detection at both /start-working and /end-working | For users with multiple GitHub accounts, `gh pr create` frequently fails because the active account does not match. The gh CLI's active account is global state and does not follow the working directory. The approach is: extract the repo owner from `git remote` → compare against `gh api /user` → on mismatch, automatically run `gh auth switch`. For single-account users this is a no-op (zero noise). SSH and direnv-level configuration is intentionally not automated, because that belongs to the user's machine-environment management and is outside the framework's responsibility |
| /release slash command | The release flow institutionalized as a slash command | When the Lead released v0.6.16 it skipped scripts/release.sh and ran git tag + git push directly on main, which was intercepted by the hook. Root cause: releasing is a ritual operation, but without a slash-command constraint the Lead had room for improvisation and would take shortcuts. Fix: just like /start-working and /end-working, give it step-by-step instructions. Principle: every ritual operation must be constrained by a slash command; the Lead's autonomy is limited to the development work itself |
| release.sh: drop the local tag push | gh release create --target main replaces git tag + git push | The local tag push runs on main and conflicts with the push-on-main hook. gh release create creates the tag via the GitHub API and bypasses the local git push entirely. The behavior is equivalent and the risk of false hook interception is eliminated |
| Expand the self-referential boundary | All framework files belong to the self-referential boundary; directories are no longer listed one by one | The old definition only covered hooks/*.sh and rules/*.json, leaving commands/*.md and scripts/*.sh outside the allowlist — so the Lead had to go through the Codex flow when editing these files, even though the actual changes (behavioral templates, small script tweaks) were not worth the Codex overhead. The expanded definition covers commands/, templates/, scripts/, hooks/, agents/, and docs/, which is consistent with the project's "framework editing its own framework" nature |
