## Framework feedback — 2026-04-09 Session #d (v0.7.5 Wave close-out bookkeeping + v0.7.5 release PO audit)

### F1 — CLAUDE.md Branch Protocol cleanup step does not specify local-vs-remote ordering; `git push origin --delete` from main is intercepted by the hook

**Observed session:** `docs/v075-closeout` (PR #184) → `/release 0.7.5` → (back on main), 2026-04-09

**Symptom:** CLAUDE.md Branch Protocol step 6 reads "Lead pushes branch → creates PR → merges to main → cleans up branch." This wording does not distinguish local branch deletion from remote branch deletion. In practice, after `gh pr merge --squash --delete-branch` returns, the local branch is already gone (`git branch -d` is implicit when the merge was initiated from the branch and `gh` switched back), but the remote ref is auto-deleted by GitHub's `--delete-branch` flag on `gh pr merge`, so in the happy path the cleanup is complete.

The gap surfaces when an operator (or a sub-agent) attempts a manual `git push origin --delete <branch>` from `main` after returning to main — the Process Observer `main-operation` hook intercepts the push as a write-to-main operation and blocks it. The correct sequence is to delete remote refs while still on the feature branch (before `git checkout main`), or to rely on `gh pr merge --delete-branch`. Neither path is spelled out in CLAUDE.md, and the hook's error message does not point to the correct alternative.

The PO audit agent for this session hit this gap when it incidentally attempted `git push origin --delete` during verification; the hook blocked it correctly, but the audit agent had to reason about why from scratch rather than reference a documented step.

**Root cause:** CLAUDE.md Branch Protocol was written assuming `gh pr merge --delete-branch` always succeeds and the `git branch -d` happens implicitly. The edge case — manual remote cleanup post-merge, or cleanup of a branch that was merged via web UI rather than `gh pr merge` — is not addressed. The hook and the documentation cover different regimes.

**Recovery this session:** No recovery needed. The happy path (`gh pr merge --squash --delete-branch` + automated `/release` branch cleanup inside `scripts/release.sh`) worked correctly. Local and remote branches for both `docs/v075-closeout` and `release/v0.7.5` are cleaned up. The gap is documentation-only.

**Proposed fix:** Add one sentence to CLAUDE.md Branch Protocol step 6, immediately after the existing cleanup clause:

> Remote branch deletion should rely on `gh pr merge --delete-branch` (automatic) or happen while still checked out on the feature branch (before `git checkout main`). `git push origin --delete` from `main` is intercepted by the Process Observer `main-operation` hook because `main` is a protected branch for write operations; switch to the feature branch first, or use `gh api repos/{owner}/{repo}/git/refs/heads/{branch} --method DELETE`.

**Priority:** Low. The happy path works; the gap only surfaces on manual cleanup edge cases or when a sub-agent probes branch state. Not a correctness or security issue — a documentation clarification.

**Affected files if accepted:**
- `CLAUDE.md` Branch Protocol section — single sentence amendment
- (Optional) `hooks/process-observer/rules/dangerous-operations.json` `main-operation` rule description — add a hint pointing at `gh pr merge --delete-branch` as the canonical cleanup path

### F2 — Doc Engineer ad-hoc fix exception and emergency hotfix exception do not cover automated `release/` branch commits generated by `scripts/release.sh`

**Observed session:** `release/v0.7.5` auto-generated by `scripts/release.sh 0.7.5`, PR #185, 2026-04-09

**Symptom:** CLAUDE.md Solo + Agent Team workflow step 4 defines two Doc Engineer bypass paths:
1. **Ad-hoc fix exception** — session does not complete any Wave AND the changes have no code↔documentation sync risk
2. **Emergency hotfix exception** — `hotfix/` branches with ≤3 changed files limited to Tier 1 `*.sh` and/or `CHANGELOG.md` during an explicit emergency release window

Neither exception explicitly names `release/` branches. The automated release commit from `scripts/release.sh` touches exactly two files (`CHANGELOG.md` date-stamp + `VERSION` bump), contains no semantic changes (date stamp + version number), and is generated mechanically by the script — yet a strict reading of step 4 would require a Doc Engineer audit for this commit because it does not match either exception wording.

In practice, the PO audit for this session classified PR #185 as N/A for C1 (Doc Engineer execution) on the reasoning that "PR #185 (`release/v0.7.5`) is fully automated by `scripts/release.sh` touching only `CHANGELOG.md` + `VERSION`; no Doc Engineer step is required or applicable for an automated release commit." The classification is correct in spirit but not formally authorized by the step 4 exceptions.

**Root cause:** Step 4's exception wording was extended in v0.7.2 to cover `hotfix/` branches explicitly and ad-hoc fix skips. The `release/` branch case was not contemplated because at the time the release flow was seen as "wrapped around" the Wave's Doc Engineer audit (Wave's DE covers the content; release only bumps version). But the release flow creates a brand-new commit on a brand-new branch that structurally reads as a fresh session requiring its own audit.

**Recovery this session:** No rework. PO audit applied the implicit reasoning and N/A'd C1 for PR #185. The two-line release commit is trivially sync-safe. The gap is rule-text-only.

**Proposed fix:** Add a third sub-bullet to CLAUDE.md Solo + Agent Team workflow step 4 (and mirror in `CLAUDE-TEMPLATE.md`):

> **Automated release exception (release/ branches only):** when a `release/vX.Y.Z` branch is created by `scripts/release.sh` and contains exactly two changed files (`CHANGELOG.md` date-stamp transition from `[Unreleased]` to `[X.Y.Z] - YYYY-MM-DD`, and `VERSION` semver bump), Doc Engineer is skipped because the commit is mechanically generated and contains no semantic content requiring sync with other documentation. The preceding PR (the Wave or fix PR being released) is responsible for its own Doc Engineer audit; the release commit rides on that audit.

**Priority:** Medium-low. Every release triggers this gap; PO audits have to reason about it each time. Writing the exception into step 4 removes the ambiguity and makes future audits mechanical.

**Affected files if accepted:**
- `CLAUDE.md` Solo + Agent Team workflow step 4 — new sub-bullet
- `CLAUDE-TEMPLATE.md` — mirror the amendment
- (Optional) `docs/workflow.md` Hotfix Workflow section — add a parallel "Automated release exception" paragraph for consistency
- (Optional) `agents/process-observer-audit.md` C1 check description — add a one-line note that automated release commits are C1 N/A by rule, not by reasoning
